# Phase 09.1: Integração Dados Meteorológicos Reais - Context

**Gathered:** 2026-04-22 (assumptions mode, --auto)
**Status:** Ready for planning

<domain>
## Phase Boundary

Augmentar a integração meteorológica atual (Open-Meteo forecast-only) com dados observados reais de duas fontes adicionais: **Visual Crossing** (precipitação medida por hora, passado e previsão) e **REDEMET METAR** (observações aeronáuticas horárias do aeroporto de Florianópolis, SBFL). O riskEngine passa a usar dados observados reais para o cálculo do fator de saturação (precip_48h_mm) quando disponíveis, mantendo Open-Meteo como fallback e fonte primária de previsão futura.

Fora do escopo: substituir completamente Open-Meteo; integração INMET direta (A806 status=Pane, TLS failures confirmados).

</domain>

<decisions>
## Implementation Decisions

### Schema — Nova Tabela de Observações
- **D-01:** Criar tabela `weather_observations` separada das `forecasts` para dados observados reais. Não alterar UNIQUE constraint de `forecasts` (atualmente `UNIQUE(bairro, forecast_time)`) para evitar quebra do INSERT OR REPLACE do forecastService. A nova tabela tem UNIQUE `(bairro, observed_time, fonte)` para suportar múltiplas fontes por timestamp.
- **D-02:** Schema da nova tabela:
  ```sql
  CREATE TABLE IF NOT EXISTS weather_observations (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    bairro          TEXT    NOT NULL DEFAULT 'florianopolis',
    observed_time   TEXT    NOT NULL,
    precipitacao_mm REAL    NOT NULL DEFAULT 0,
    fonte           TEXT    NOT NULL,  -- 'visual-crossing' | 'redemet-metar'
    fetched_at      TEXT    NOT NULL DEFAULT (datetime('now')),
    UNIQUE(bairro, observed_time, fonte)
  )
  ```
- **D-03:** A tabela `forecasts` e o `forecastService.js` permanecem intactos — Open-Meteo continua funcionando exatamente como está.

### Fontes de Dados e Seus Papéis
- **D-04:** **Visual Crossing** — fonte primária para precipitação medida (mm) das últimas 48h. Endpoint: `/VisualCrossingWebServices/rest/services/timeline/{lat},{lon}/{date1}/{date2}?unitGroup=metric&include=hours&key={KEY}`. Credencial: `VISUAL_CROSSING_API_KEY` no `.env`. Frequência: 1 vez por hora, mesma janela do forecastService.
- **D-05:** **REDEMET METAR** — observações aeronáuticas do SBFL (ICAO: `SBFL`). Fornece tipo de precipitação (RA, TSRA, etc.) e intensidade qualitativa. Endpoint REDEMET API: `https://api.redemet.aer.mil.br/mensagens/metar/SBFL`. Credencial: `REDEMET_API_KEY` no `.env`. Extrair mm de precipitação do campo METAR quando disponível; caso contrário registrar 0 com flag qualitativa.
- **D-06:** **Open-Meteo past_hours** — já implementado no forecastService.js (parâmetro `past_hours=48`). Continua como fallback quando Visual Crossing indisponível ou sem API key.
- **D-07:** **INMET** — não integrar. Estação A806 (Florianópolis) status="Pane". TLS failures confirmados. O arquivo `inmet_estacoes_sc.json` na raiz do backend serve apenas como referência de catálogo; não usá-lo em código de produção.

### Degradação Graceful — API Keys Ausentes
- **D-08:** Se `VISUAL_CROSSING_API_KEY` não definida no `.env`: logar aviso `[visual-crossing] API key ausente — fonte ignorada` e pular o fetch. Sem throw, sem impacto no forecast Open-Meteo.
- **D-09:** Se `REDEMET_API_KEY` não definida: mesmo padrão — logar e pular. Sistema continua operando com Open-Meteo como única fonte.
- **D-10:** Ambos os serviços seguem o padrão no-throw de `forecastService.js`: em caso de erro de rede ou parse, logar e retornar sem atualizar `weather_observations`. Dados stale permanecem válidos.

### riskEngine — Integração com Dados Reais
- **D-11:** O cálculo de `precip_48h_mm` no `forecastService.js` (salvo em `forecasts_meta`) deve ser complementado com dado observado. Estratégia: após busca do Visual Crossing, atualizar `forecasts_meta.precip_48h_mm` com a soma das observações reais das últimas 48h se disponíveis; caso contrário manter valor Open-Meteo.
- **D-12:** As janelas de previsão futura no `riskEngine.js` (24h/48h/72h à frente) continuam usando apenas a tabela `forecasts` (Open-Meteo forecast). Não misturar observações passadas com previsões futuras na mesma query.
- **D-13:** O campo `precipitacao_prevista_mm` em `risk_scores` continua representando precipitação futura prevista (Open-Meteo). Renomear ou adicionar campo de precipitação observada fica para versão futura.

### Scheduler — Novos Jobs
- **D-14:** Dois novos módulos de serviço: `visualCrossingService.js` e `redemetService.js`, ambos em `backend/src/services/`.
- **D-15:** Registrar ambos no `scheduler.js` existente com o mesmo intervalo de 1h do `forecastService`. Ordem de execução: forecastService → visualCrossingService → redemetService → calcularRiscos.
- **D-16:** No startup (`initScheduler()`): executar os três fetches na sequência antes de ativar o cron, igual ao padrão atual do forecastService.

### Claude's Discretion
- Parsing exato do METAR para extrair mm de precipitação (campos RMK, P001 etc.) — pode ser que METAR não forneça mm precisos, nesse caso registrar 0 e log descritivo
- Janela exata de busca do Visual Crossing (últimas 48h ou últimas 24h)
- Estratégia de merge quando Visual Crossing e REDEMET têm dados para o mesmo timestamp

</decisions>

<specifics>
## Specific Ideas

- Manter Open-Meteo completamente intacto — novos serviços são aditivos, não substitutivos
- INMET A806 está "Pane" (offline) — não tentar integrar INMET mesmo que o arquivo de estações esteja no repo
- Se REDEMET METAR não fornecer mm precisos, registrar 0 e logar o tipo de precipitação (RA, TSRA) para debug futuro
- Visual Crossing é a fonte mais confiável para mm observados — priorizar sobre REDEMET para atualizar `precip_48h_mm`

</specifics>
