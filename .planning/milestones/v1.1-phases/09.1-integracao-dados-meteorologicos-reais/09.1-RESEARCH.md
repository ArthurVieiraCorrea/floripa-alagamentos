## Summary

Esta fase adiciona duas fontes de dados observados reais ao sistema: Visual Crossing Timeline API (precipitação horária em mm, últimas 48h) e REDEMET METAR SBFL (observações aeronáuticas qualitativas). A pesquisa confirmou os formatos exatos de ambas as APIs, com uma descoberta crítica: **METARs brasileiros não incluem o P-group (mm de precipitação) na seção RMK** — essa é uma convenção exclusiva de estações NWS/FAA norte-americanas. Portanto, REDEMET fornece apenas informação qualitativa (RA, TSRA, DZ) e deve registrar `precipitacao_mm = 0` com um log do `mens` original. Visual Crossing permanece como única fonte de mm observados.

O custo de API do Visual Crossing é gerenciável: 48 registros/hora para uma query de 48h horária (1 record = 1 hora × 1 localidade). Com 24 chamadas/dia, o consumo é 24 × 48 = 1.152 records/dia — ligeiramente acima do free tier de 1.000/dia. A janela deve ser reduzida para **últimas 24h** (24 records/call × 24 calls/day = 576 records/day) para caber no free tier, ou usar `last2days` com 48 records mas fazer apenas uma chamada se os dados já estiverem em cache e forem recentes.

**Primary recommendation:** Usar janela de 24h no Visual Crossing (não 48h) para caber confortavelmente no free tier; aceitar que REDEMET contribui apenas dados qualitativos sem mm; o `precip_48h_mm` em `forecasts_meta` deve somar Visual Crossing (24h observado) + Open-Meteo (24h futuro) como estratégia híbrida mais robusta.

### Pattern 1: Service Module — Clonar forecastService.js

**What:** Cada novo serviço replica exatamente o padrão de `forecastService.js`: `fetchWithRetry()` privado, `fetchAndCacheObservations()` exportado, no-throw, AbortController timeout 10s, transação SQLite com rollback.

**When to use:** Para qualquer novo serviço de dados externos neste projeto.

**Exemplo — visualCrossingService.js (estrutura):**
```javascript
// Source: padrão de forecastService.js existente
'use strict';
const { getDb } = require('../config/database');

const FETCH_TIMEOUT_MS = 10_000;
const MAX_RETRIES = 3;
const RETRY_DELAY_MS = 5_000;

function sleep(ms) { return new Promise(r => setTimeout(r, ms)); }

async function fetchWithRetry(url) {
  let lastErr;
  for (let attempt = 1; attempt <= MAX_RETRIES; attempt++) {
    const controller = new AbortController();
    const tid = setTimeout(() => controller.abort(), FETCH_TIMEOUT_MS);
    try {
      const res = await fetch(url, { signal: controller.signal });
      clearTimeout(tid);
      if (!res.ok) throw new Error(`Visual Crossing HTTP ${res.status}`);
      return await res.json();
    } catch (err) {
      clearTimeout(tid);
      lastErr = err;
      if (attempt < MAX_RETRIES) {
        const wait = RETRY_DELAY_MS * Math.pow(2, attempt - 1);
        console.warn(`[visual-crossing] Tentativa ${attempt}/${MAX_RETRIES} falhou: ${err.message}. Aguardando ${wait/1000}s...`);
        await sleep(wait);
      }
    }
  }
  throw lastErr;
}

async function fetchAndCacheObservations() {
  if (!process.env.VISUAL_CROSSING_API_KEY) {
    console.warn('[visual-crossing] API key ausente — fonte ignorada');
    return;
  }
  // ... fetch + INSERT OR REPLACE + UPDATE forecasts_meta
}

module.exports = { fetchAndCacheObservations };
```

### Pattern 2: Visual Crossing — Endpoint e Resposta

**Endpoint:**
```
GET https://weather.visualcrossing.com/VisualCrossingWebServices/rest/services/timeline/-27.5954,-48.5480/yesterday/today?unitGroup=metric&include=hours&elements=datetime,precip,preciptype&key={KEY}
```

**Parâmetros:**
| Parâmetro | Valor | Motivo |
|-----------|-------|--------|
| `location` | `-27.5954,-48.5480` | Coordenadas de Florianópolis (mesmas do Open-Meteo) |
| `date1` | `yesterday` | Dinâmico — ontem |
| `date2` | `today` | Dinâmico — hoje (janela de ~24-48h) |
| `unitGroup` | `metric` | Retorna `precip` em mm |
| `include` | `hours` | Dados horários (não diários) |
| `elements` | `datetime,precip,preciptype` | Reduz payload; só o necessário |
| `key` | `{VISUAL_CROSSING_API_KEY}` | Autenticação |

**Querido (não verificado empiricamente):** `yesterday/today` abrange aproximadamente 48h. Usar datas calculadas explicitamente (`YYYY-MM-DD`) é mais confiável:
```javascript
// Calcular yesterday e today em horário local (UTC-3)
const now = new Date();
const yesterday = new Date(now.getTime() - 24 * 60 * 60 * 1000);
const fmt = d => d.toLocaleDateString('sv-SE'); // 'sv-SE' = YYYY-MM-DD
const url = `https://weather.visualcrossing.com/.../timeline/${LAT},${LON}/${fmt(yesterday)}/${fmt(now)}?unitGroup=metric&include=hours&elements=datetime,precip,preciptype&key=${KEY}`;
```

**Estrutura de resposta JSON:**
```json
{
  "queryCost": 48,
  "latitude": -27.5954,
  "longitude": -48.548,
  "timezone": "America/Sao_Paulo",
  "days": [
    {
      "datetime": "2026-04-22",
      "hours": [
        {
          "datetime": "00:00:00",
          "precip": 0.0,
          "preciptype": null
        },
        {
          "datetime": "01:00:00",
          "precip": 2.3,
          "preciptype": ["rain"]
        }
      ]
    }

```

**Campo-chave:** `days[i].hours[j].precip` — precipitação em mm quando `unitGroup=metric`. Pode ser `null` (tratar como 0).

**`observed_time` para INSERT:** Combinar `days[i].datetime` + `hours[j].datetime` para formar timestamp ISO local: `"2026-04-22T01:00:00"`.

### Pattern 3: REDEMET METAR — Endpoint e Resposta

**Endpoint verificado (curl confirmou reachability):**
```
GET https://api-redemet.decea.mil.br/mensagens/metar/SBFL?api_key={KEY}&data_ini={YYYYMMDDHH}&data_fim={YYYYMMDDHH}
```

**Parâmetros:**
| Parâmetro | Valor | Notas |
|-----------|-------|-------|
| `localidades` | `SBFL` | ICAO code Florianópolis |
| `api_key` | `{REDEMET_API_KEY}` | Query param (confirmado) |
| `data_ini` | `YYYYMMDDHH` | Hora UTC de início (ex: `2026042200`) |
| `data_fim` | `YYYYMMDDHH` | Hora UTC de fim (ex: `2026042223`) |

**Sem `data_ini`/`data_fim`:** Retorna o METAR mais recente disponível (útil para fetch horário simples).

**Estrutura de resposta JSON (confirmada via documentação oficial):**
```json
{
  "status": true,
  "message": 200,
  "data": {
    "current_page": 1,
    "data": [
      {
        "id_localidade": "SBFL",
        "validade_inicial": "2026-04-23 14:00:00",
        "mens": "METAR SBFL 231400Z 12008KT 9999 -RA FEW025 SCT060 22/19 Q1015=",
        "recebimento": "2026-04-23 13:57:00"
      }
    ],
    "per_page": 150,
    "total": 1,
    "last_page": 1
  }
}
```

**Campo-chave:** `data.data[].mens` — string METAR completa. Extrair timestamp de `validade_inicial`.

**DESCOBERTA CRÍTICA: Brazilian METARs não contêm P-group de precipitação em mm.**

O P-group (`P0015` = 0.15 polegadas de chuva na última hora) é uma adição exclusiva de estações NWS/ASOS dos EUA/Canadá. Estações ICAO internacionais, incluindo todas as brasileiras (DECEA/REDEMET), seguem o padrão ICAO Annex 3 que **não inclui** este grupo nas observações padrão.

O que o METAR SBFL PODE conter:
- **Presente no grupo de tempo:** `-RA` (chuva fraca), `RA` (chuva moderada), `+RA` (chuva forte), `TSRA` (trovoada com chuva), `DZ` (chuvisco), `SN` (neve — raro em Floripa)
- **Grupo `RERA`/`REOBSC` em RMK:** Fenômeno recente (ex: `RERA` = chuva recente)
- **Sem mm quantitativos** — nunca `P0015` nem similar em SBFL

**Estratégia de parsing METAR (Claude's Discretion):**
```javascript
// Extrair tipo qualitativo de precipitação do campo de tempo presente
// Regex para grupos de tempo presente padrão ICAO
function parsePrecipFromMETAR(mens) {
  // Tempo presente: -RA, RA, +RA, TSRA, -DZ, DZ, +DZ, RASN, etc.
  // Padrão: (VC|MI|PR|BC|DR|BL|FZ|SH|TS|+|-)?  seguido de fenômeno
  const precipMatch = mens.match(/\b(-|\+)?(VC)?(TS|SH|FZ|BL|DR|BC|PR|MI)?(DZ|RA|SN|SG|IC|PL|GR|GS|UP)(\b|TS)/);
  if (precipMatch) {
    return {
      precipitacao_mm: 0,  // sem dado quantitativo para METAR internacional
      tipo_qualitativo: precipMatch[0]  // ex: "-RA", "TSRA", "+RA"
    };
  }
  return { precipitacao_mm: 0, tipo_qualitativo: null };
}
```

**Nota:** `precipitacao_mm = 0` em TODAS as inserções REDEMET (sem dado quantitativo). O valor do serviço REDEMET é:
1. Confirmar presença/ausência de precipitação nas últimas horas
2. Fornecer dado de qualidade aeronáutica como referência de validação
3. Servir como sinal de segunda fonte para lógica futura

### Pattern 4: riskEngine.js — Leitura de precip_48h_mm

**Modificação (D-11):** Após Visual Crossing popular `weather_observations`, `forecasts_meta.precip_48h_mm` deve ser atualizado com a soma observada real. O `riskEngine.js` em si não precisa mudar — ele já lê de `forecasts_meta.precip_48h_mm`. A atualização ocorre dentro de `visualCrossingService.fetchAndCacheObservations()`.

```javascript
// Em visualCrossingService.js, após inserir observações:
const row = db.get(
  `SELECT COALESCE(SUM(precipitacao_mm), 0) AS total_mm
     FROM weather_observations
    WHERE bairro = 'florianopolis'
      AND fonte = 'visual-crossing'
      AND observed_time >= datetime('now', '-3 hours', '-24 hours')
      AND observed_time <  datetime('now', '-3 hours')`
);
const observado_mm = Math.round((row.total_mm ?? 0) * 10) / 10;

db.run(
  `UPDATE forecasts_meta SET precip_48h_mm = ? WHERE id = 1`,
  [observado_mm]
);
console.log(`[visual-crossing] precip_48h_mm atualizado: ${observado_mm}mm (observado real)`);
```

**Nota sobre janela:** O campo se chama `precip_48h_mm` mas a janela Visual Crossing é de 24h (para caber no free tier). O riskEngine usa esse campo como sinal de saturação do solo — 24h de dado real é melhor que 48h estimado. Se o planner decidir usar janela de 48h (pagando acima do free tier ou com cache inteligente), a query acima ajusta facilmente.

### Anti-Patterns to Avoid

- **Não usar `/last2days` como shortcut de URL:** A documentação confirmou que o shortcut existe, mas calcular `yesterday`/`today` explicitamente com `YYYY-MM-DD` é mais previsível, especialmente em torno da meia-noite e mudanças de dia.
- **Não misturar `validade_inicial` (UTC) com `observed_time` (local):** O campo `validade_inicial` do REDEMET é UTC. Converter para UTC-3 antes de armazenar em `observed_time` para consistência com o resto do banco.
- **Não reusar `forecast_time` como `observed_time`:** As tabelas têm semânticas diferentes — `forecasts.forecast_time` é hora local sem offset; `weather_observations.observed_time` deve seguir o mesmo padrão para que queries `datetime('now', '-3 hours')` funcionem corretamente.
- **Não fazer throw em erros de API externa:** Seguir padrão no-throw de `forecastService.js` — logar e retornar.
- **Não chamar REDEMET sem `data_ini`/`data_fim` para histórico:** Sem parâmetros de data, o endpoint retorna apenas o METAR mais recente; para backfill das últimas horas, passar janela explícita.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP retry com backoff | Retry loop customizado | Copiar `fetchWithRetry()` de `forecastService.js` | Já testado em produção; inclui AbortController + backoff exponencial |
| METAR string parser completo | Parser METAR completo | Regex simples para grupo de tempo presente | SBFL não inclui dados quantitativos em RMK; parser completo é desperdício de complexidade |
| Date formatting YYYY-MM-DD | `moment.js` ou `date-fns` | `d.toLocaleDateString('sv-SE')` (nativo) | Locale `sv-SE` produz formato ISO YYYY-MM-DD nativamente |
| SQLite upsert | UPDATE + INSERT separados | `INSERT OR REPLACE` (já usado) | UNIQUE constraint garante idempotência |

**Key insight:** O projeto deliberadamente evita dependências extras. Toda a lógica necessária existe em `forecastService.js` como template.

## Common Pitfalls

### Pitfall 1: P-group METAR ausente em estações brasileiras
**What goes wrong:** Código tenta parsear `P\d{4}` no RMK section do METAR SBFL e nunca encontra nada, resultando em `precipitacao_mm = 0` para todas as inserções REDEMET — sem erro explícito, comportamento silenciosamente incorreto.
**Why it happens:** O P-group é uma adição NWS/FAA norte-americana; documentação genérica de METAR menciona ele, mas estações ICAO internacionais (incluindo Brasil/DECEA) não o incluem.
**How to avoid:** Aceitar desde o início que REDEMET fornece apenas dado qualitativo. Registrar `precipitacao_mm = 0` sempre para esta fonte; logar o `tipo_qualitativo` extraído do grupo de tempo presente para rastreabilidade.
**Warning signs:** Se os logs mostrarem 0mm para todos os registros REDEMET mesmo em dias de chuva — isso é o comportamento esperado, não um bug.

### Pitfall 2: Custo excessivo no free tier do Visual Crossing
**What goes wrong:** Query de 48h horária = 48 records/call × 24 calls/day = 1.152 records/day. O free tier é 1.000 records/day. Em 1 dia, o limite é ultrapassado; a API retorna HTTP 429.
**Why it happens:** 1 hora = 1 record para dados históricos horários.
**How to avoid:** Usar janela de 24h (`yesterday` + `today`) = 24 records/call × 24 calls/day = 576 records/day (< 1.000). Ou verificar `queryCost` na resposta e logar aviso quando > 30.
**Warning signs:** Response body com `{"message":"Daily quota exceeded"}` ou HTTP 429.

### Pitfall 3: Timezone mismatch entre REDEMET (UTC) e banco (UTC-3 local)
**What goes wrong:** `validade_inicial: "2026-04-23 14:00:00"` é UTC. Inserir diretamente como `observed_time` sem converter cria um offset de 3h em relação ao resto do banco.
**Why it happens:** REDEMET retorna timestamps em UTC; o restante do sistema usa hora local Florianópolis (UTC-3, sem DST).
**How to avoid:** Ao processar cada METAR, converter `validade_inicial` de UTC para UTC-3 antes de inserir em `observed_time`.
```javascript
// Converter UTC → UTC-3 (Florianópolis, sem DST)
const utcTime = new Date(row.validade_inicial.replace(' ', 'T') + 'Z'); // força UTC
const localOffset = -3 * 60 * 60 * 1000; // UTC-3 em ms
const localTime = new Date(utcTime.getTime() + localOffset);
const observed_time = localTime.toISOString().replace('T', ' ').substring(0, 19);
```

### Pitfall 4: Visual Crossing datetime parsing — combinação de data + hora
**What goes wrong:** `days[i].datetime = "2026-04-22"` e `hours[j].datetime = "01:00:00"`. Código tenta usar cada um isolado, ou combina errado como `"2026-04-22 01:00:00"` sem considerar que é hora local já.
**Why it happens:** A API retorna data no nível de `days` e hora no nível de `hours` como campos separados.
**How to avoid:** Concatenar explicitamente: `observed_time = days[i].datetime + 'T' + hours[j].datetime.substring(0, 5)` → `"2026-04-22T01:00"`. Já está em hora local (timezone configurado no request).

### Pitfall 5: REDEMET sem `data_ini`/`data_fim` retorna apenas o último METAR
**What goes wrong:** Sem parâmetros de data, a API retorna o METAR mais recente (1 registro). Se o serviço cair e voltar 3h depois, os 3 METARs intermediários são perdidos.
**Why it happens:** Comportamento default da API sem parâmetros de janela.
**How to avoid:** Sempre passar `data_ini` = última hora inteira e `data_fim` = hora atual para capturar todos os METARs desde o último fetch. Para o caso simples (fetch a cada 1h), `data_ini = data_fim = hora atual UTC (YYYYMMDDHH)` é suficiente.

### Pitfall 6: INSERT OR REPLACE destrói `fetched_at` anterior em rows existentes
**What goes wrong:** `INSERT OR REPLACE` deleta a row existente e insere nova, perdendo o `fetched_at` original.
**Why it happens:** SQLite trata `INSERT OR REPLACE` como DELETE + INSERT quando há conflito UNIQUE.
**How to avoid:** Usar `INSERT OR IGNORE` para observações (dados históricos são imutáveis — se já existe um registro para o mesmo `(bairro, observed_time, fonte)`, não precisa sobrescrever). Ou `INSERT OR REPLACE` está OK se apenas `fetched_at` muda (aceitável para este caso).

### Visual Crossing — Parse response e INSERT
```javascript
// Source: padrão forecastService.js + estrutura confirmada da documentação VC
const data = await fetchWithRetry(url);
const db = getDb();
let totalMm = 0;

db.run('BEGIN');
try {
  for (const day of data.days ?? []) {
    for (const hour of day.hours ?? []) {
      const mm = hour.precip ?? 0;
      // Combinar data + hora: "2026-04-22" + "T" + "01:00:00".slice(0,5) = "2026-04-22T01:00"
      const observed_time = `${day.datetime}T${hour.datetime.substring(0, 5)}`;
      totalMm += mm;

      db.run(
        `INSERT OR IGNORE INTO weather_observations
           (bairro, observed_time, precipitacao_mm, fonte, fetched_at)
         VALUES ('florianopolis', ?, ?, 'visual-crossing', datetime('now'))`,
        [observed_time, mm]
      );
    }
  }
  db.run('COMMIT');
} catch (err) {
  try { db.run('ROLLBACK'); } catch (_) {}
  throw err;
}

// Atualizar forecasts_meta.precip_48h_mm com dado observado real
const rounded = Math.round(totalMm * 10) / 10;
db.run(
  `UPDATE forecasts_meta SET precip_48h_mm = ? WHERE id = 1`,
```

```javascript
// Source: https://ajuda.decea.mil.br/base-de-conhecimento/api-redemet-mensagem-metar/
```

### REDEMET — Parse e INSERT
```javascript
// Source: documentação DECEA + descoberta sobre P-group ausente em METARs brasileiros
const data = await fetchWithRetry(url);
if (!data?.status) {
  console.warn('[redemet] Resposta inesperada:', JSON.stringify(data).slice(0, 200));
  return;
}

const registros = data?.data?.data ?? [];
if (registros.length === 0) {
  console.log('[redemet] Nenhum METAR disponível para este período');
  return;
}

const db = getDb();
db.run('BEGIN');
try {
  for (const registro of registros) {
    const mens = registro.mens ?? '';

    // METAR brasileiro NÃO contém P-group (mm) — apenas informação qualitativa
    // Extrair tipo de precipitação do grupo de tempo presente
    const precipMatch = mens.match(/\s(-|\+)?(VC)?(TS|SH|FZ)?(DZ|RA|SN|SG|IC|PL|GR|GS)/);
    const tipoQualitativo = precipMatch ? precipMatch[0].trim() : null;

    // Converter validade_inicial (UTC) para hora local UTC-3
    const utcTime = new Date(registro.validade_inicial.replace(' ', 'T') + 'Z');
    const localMs = utcTime.getTime() - 3 * 60 * 60 * 1000;
    const localTime = new Date(localMs);
    const observed_time = localTime.toISOString().replace('T', ' ').substring(0, 16)
      .replace(' ', 'T'); // "2026-04-23T11:00"

    db.run(
      `INSERT OR IGNORE INTO weather_observations
         (bairro, observed_time, precipitacao_mm, fonte, fetched_at)
       VALUES ('florianopolis', ?, 0, 'redemet-metar', datetime('now'))`,
      [observed_time]
    );

    if (tipoQualitativo) {
      console.log(`[redemet] SBFL ${observed_time}: ${tipoQualitativo} (qualitativo; sem mm)`);
    }
  }
  db.run('COMMIT');
} catch (err) {
  try { db.run('ROLLBACK'); } catch (_) {}
  throw err;
}
```

## Open Questions

1. **Janela de 24h vs 48h para Visual Crossing**
   - O que sabemos: 24h/call × 24 calls/day = 576 records/day (< 1.000 free tier). 48h/call × 24 calls/day = 1.152 records/day (excede free tier).
   - O que está indefinido: o riskEngine usa `precip_48h_mm` como nome — mas o planner pode decidir usar 24h observado + fallback Open-Meteo para as horas mais antigas.
   - Recomendação: usar 24h por padrão; nomear variável interna `precip_obs_mm` mas gravar em `precip_48h_mm` para compatibilidade com riskEngine.

2. **REDEMET timezone de `validade_inicial`**
   - O que sabemos: endpoint confirmado ativo; resposta JSON estruturada. A documentação não menciona explicitamente se `validade_inicial` é UTC ou local.
   - O que está indefinido: timezone exato do campo.
   - Recomendação: tratar como UTC (padrão internacional para aviação), logar na primeira execução para confirmar.

3. **Comportamento do REDEMET sem chave de API (graceful degradation)**
   - O que sabemos: resposta sem chave é `{"error": {"code": "API_KEY_MISSING"}}` com HTTP 200 (!) ou 401 — curl confirmou JSON response mas não verificamos o HTTP status code.
   - O que está indefinido: o status code exato (400/401/403) para o guard de API key ausente.
   - Recomendação: verificar `!process.env.REDEMET_API_KEY` antes de fazer o fetch (D-09), independente do status code.
