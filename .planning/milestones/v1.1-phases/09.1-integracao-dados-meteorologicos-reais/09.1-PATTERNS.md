## File 1: `backend/src/services/visualCrossingService.js` (CREATE)

### Role
Fetches hourly precipitation (mm) for Florianópolis from Visual Crossing Timeline API for the last ~48h (yesterday + today), persists rows to `weather_observations`, and updates `forecasts_meta.precip_48h_mm` with the real observed total. This is the only service that writes a non-zero `precipitacao_mm` to `weather_observations`.

### URL Construction Pattern

```javascript
// From RESEARCH.md Code Examples — verified against Visual Crossing Timeline API docs
const LAT = -27.5954;
const LON = -48.5480;

function buildUrl() {
  const KEY = process.env.VISUAL_CROSSING_API_KEY;
  const now = new Date();
  const yesterday = new Date(now.getTime() - 24 * 60 * 60 * 1000);
  // toLocaleDateString('sv-SE') produces YYYY-MM-DD natively (no external dep)
  const toLocalDate = d => {
    const local = new Date(d.getTime() - 3 * 60 * 60 * 1000); // UTC-3
    return local.toISOString().substring(0, 10);
  };
  const date1 = toLocalDate(yesterday);
  const date2 = toLocalDate(now);
  return [
    'https://weather.visualcrossing.com/VisualCrossingWebServices/rest/services/timeline',
    `/${LAT},${LON}/${date1}/${date2}`,
    '?unitGroup=metric',
    '&include=hours',
    '&elements=datetime,precip,preciptype',
    `&key=${KEY}`
  ].join('');
}
```

### Response Parsing Pattern

```javascript
// data.days is an array of day objects; each day has data.days[i].hours array
// Combine day-level datetime with hour-level datetime to form ISO local timestamp
for (const day of data.days ?? []) {
  for (const hour of day.hours ?? []) {
    const mm = hour.precip ?? 0;
    // "2026-04-22" + "T" + "01:00:00".substring(0, 5) → "2026-04-22T01:00"
    const observed_time = `${day.datetime}T${hour.datetime.substring(0, 5)}`;
    totalMm += mm;
    db.run(
      `INSERT OR IGNORE INTO weather_observations
         (bairro, observed_time, precipitacao_mm, fonte, fetched_at)
       VALUES ('florianopolis', ?, ?, 'visual-crossing', datetime('now'))`,
      [observed_time, mm]
    );
  }
}
```

## File 2: `backend/src/services/redemetService.js` (CREATE)

### Role
Fetches the current hour's METAR message for SBFL from REDEMET API. Persists one row per METAR to `weather_observations` with `precipitacao_mm = 0` always (Brazilian METARs carry no P-group quantitative data — see RESEARCH.md Pitfall #1). Value is qualitative: precipitation type (`-RA`, `TSRA`, etc.) is logged for traceability.

### Analog: `forecastService.js` (same structure as visualCrossingService.js)

Replicate:
- Same module structure, constants, `sleep()`, `fetchWithRetry(url)` signature
- Same no-throw outer wrapper
- Same `db.run('BEGIN')` / `db.run('COMMIT')` / `db.run('ROLLBACK')` transaction

### Key Differences

| Aspect | visualCrossingService.js | redemetService.js |
|--------|--------------------------|-------------------|
| Guard | `VISUAL_CROSSING_API_KEY` | `REDEMET_API_KEY` |
| Log prefix | `[visual-crossing]` | `[redemet]` |
| fonte value | `'visual-crossing'` | `'redemet-metar'` |
| `precipitacao_mm` | Real mm from API | Always `0` |
| Response shape | `data.days[].hours[]` | `data.data.data[]` (nested) |
| Timestamp source | `day.datetime + hour.datetime` | `registro.validade_inicial` converted UTC → UTC-3 |
| `forecasts_meta` update | Yes — updates `precip_48h_mm` | No — REDEMET does not update `precip_48h_mm` |
| No-data path | `data.days` empty | `data.data.data` empty — log and return |

### URL Construction Pattern

```javascript
// REDEMET uses UTC timestamps as YYYYMMDDHH string params
// Without data_ini/data_fim it returns only the latest METAR (not useful for history)
function buildUrl() {
  const KEY = process.env.REDEMET_API_KEY;
  const nowUtc = new Date();
  const pad = n => String(n).padStart(2, '0');
  const dataHora = [
    nowUtc.getUTCFullYear(),
    pad(nowUtc.getUTCMonth() + 1),
    pad(nowUtc.getUTCDate()),
    pad(nowUtc.getUTCHours())
  ].join(''); // e.g. "2026042314"
  return `https://api-redemet.decea.mil.br/mensagens/metar/SBFL?api_key=${KEY}&data_ini=${dataHora}&data_fim=${dataHora}`;
}
```

### Response Shape

```javascript
// data.status === true means success
// data.data.data is the array of METAR records
// Each record: { id_localidade, validade_inicial, mens, recebimento }
const registros = data?.data?.data ?? [];
if (registros.length === 0) {
  console.log('[redemet] Nenhum METAR disponível para este período');
  return;
}
```

### UTC → UTC-3 Conversion Pattern

```javascript
// validade_inicial format: "2026-04-23 14:00:00" (assumed UTC per aviation standard ICAO)
// Convert to UTC-3 before storing as observed_time
const utcTime = new Date(registro.validade_inicial.replace(' ', 'T') + 'Z');
const localMs = utcTime.getTime() - 3 * 60 * 60 * 1000;
const observed_time = new Date(localMs).toISOString().substring(0, 16).replace('T', 'T');
// Result: "2026-04-23T11:00"
```

### Precipitation Parsing Pattern

```javascript
// Brazilian METARs do NOT include P-group (mm) — only qualitative present weather group
// Regex matches: -RA, RA, +RA, TSRA, -DZ, DZ, RASN, etc.
const precipMatch = mens.match(/\s(-|\+)?(VC)?(TS|SH|FZ)?(DZ|RA|SN|SG|IC|PL|GR|GS)/);
const tipoQualitativo = precipMatch ? precipMatch[0].trim() : null;

// INSERT always uses precipitacao_mm = 0
db.run(
  `INSERT OR IGNORE INTO weather_observations
     (bairro, observed_time, precipitacao_mm, fonte, fetched_at)
   VALUES ('florianopolis', ?, 0, 'redemet-metar', datetime('now'))`,
  [observed_time]
);
if (tipoQualitativo) {
  console.log(`[redemet] SBFL ${observed_time}: ${tipoQualitativo} (qualitativo; sem mm)`);
}
```

### What Changes

After that update, the next `calcularRiscos()` run will use the updated value naturally — because `calcularRiscos()` reads `forecasts` rows (future precipitation) while `precip_48h_mm` is a separate stored field updated by whichever fetch ran most recently.

**Verify (no code change needed):** The `precipPorJanela` query at lines 51–60 of riskEngine.js queries the `forecasts` table with `forecast_time >= datetime('now', '-3 hours')`. This covers FUTURE forecast hours from Open-Meteo. The `precip_48h_mm` field in `forecasts_meta` is used separately to represent observed/past precipitation. These two concerns do not conflict.

## Cross-File Data Flow

```
scheduler.js (hourly cron)
  │
  ├─ fetchAndCacheForecasts()          → writes forecasts table
  │                                      writes forecasts_meta (INSERT OR REPLACE id=1)
  │
  ├─ fetchVisualCrossing()             → writes weather_observations (fonte='visual-crossing')
  │                                      UPDATE forecasts_meta SET precip_48h_mm=? WHERE id=1
  │
  └─ fetchRedemet()                    → writes weather_observations (fonte='redemet-metar', mm=0)

scheduler.js (4h cron)
  └─ calcularRiscos()                  → reads forecasts (future precip by window)
                                         reads ocorrencias (historical count)
                                         writes risk_scores
```

`forecasts_meta.precip_48h_mm` is the handoff point:
- `forecastService.js` writes it first (Open-Meteo past_hours sum)
- `visualCrossingService.js` overwrites it with real observed mm (when key present)
- `riskEngine.js` reads it via `forecasts_meta` (already in existing code — no change needed)
