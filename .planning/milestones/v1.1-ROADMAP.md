# Milestone v1.1: Resiliência & UX

**Status:** ✅ SHIPPED 2026-04-28
**Phases:** 8, 9, 09.1 (INSERTED), 10
**Total Plans:** 11

## Overview

Milestone de consolidação pós-MVP. Foco em resiliência do backend contra falhas de rede, melhoria de experiência do usuário com novos painéis e histórico, integração de fontes de dados meteorológicos reais (Visual Crossing + REDEMET), e guia de onboarding para novos usuários.

## Phases

### Phase 8: Backend Resilience

**Goal**: O backend tolera falhas transitórias do Open-Meteo sem perder dados e permite ao admin sincronizar riscos imediatamente após importação CSV
**Depends on**: Phase 7 (v1.0 complete)
**Requirements**: RESIL-01, RESIL-02
**Plans**: 2 plans

Plans:
- [x] 08-01: Backoff exponencial em fetchWithRetry() (RESIL-02)
- [x] 08-02: Auto-trigger recálculo em confirmar() + checkAndSendAlerts em recalcular() + frontend cleanup (RESIL-01)

**Details:**

08-01: fetchWithRetry() atualizado de delay fixo 5s para backoff exponencial (5s → 10s → throw). Reduz pressão sobre Open-Meteo em falhas sustentadas. Log atualizado para exibir `waitMs` real calculado.

08-02: `confirmar()` tornado async — encadeia `calcularRiscos()` + `checkAndSendAlerts()` em best-effort try/catch após importação CSV. `recalcular()` também encadeia `checkAndSendAlerts()`. Botão "Recalcular" removido do admin UI (endpoint preservado em api.js via curl).

### Phase 09.1: Integração Dados Meteorológicos Reais (INSERTED)

**Goal:** O riskEngine usa precipitação observada real (mm) do Visual Crossing e observações qualitativas METAR do REDEMET para calcular precip_48h_mm, mantendo Open-Meteo como fallback quando as API keys estiverem ausentes
**Depends on**: Phase 09
**Requirements**: D-01 a D-16 (ver 09.1-CONTEXT.md)
**Plans**: 4 plans

Plans:
- [x] 09.1-01: Schema weather_observations + variáveis de ambiente (Wave 1)
- [x] 09.1-02: visualCrossingService.js — fetch + cache precipitação observada mm (Wave 2)
- [x] 09.1-03: redemetService.js — fetch + cache METAR SBFL qualitativo (Wave 2, paralelo)
- [x] 09.1-04: scheduler.js — integrar os dois novos serviços no ciclo horário (Wave 3)

**Details:**

09.1-01: DDL `weather_observations` com UNIQUE(bairro, observed_time, fonte) para multi-fonte sem conflito. Declaração de VISUAL_CROSSING_API_KEY e REDEMET_API_KEY em .env.example com degradação graceful quando ausentes.

09.1-02: visualCrossingService.js — Timeline API da Visual Crossing para precipitação observada (mm) das últimas 24h. Cache em weather_observations com INSERT OR REPLACE. Fix CR-01 aplicado: janela de datas usa dia-calendário anterior ao invés de wall-clock -24h.

09.1-03: redemetService.js — METAR SBFL via REDEMET API. Leitura qualitativa (TS, RA, SHRA, RASN). Mapeamento para valores mm estimados. Cache na mesma tabela weather_observations.

09.1-04: scheduler.js integrado — ciclo horário agora chama visualCrossing + REDEMET em paralelo antes de fetchAndCacheForecasts(). riskEngine atualizado para usar precip_48h_mm de weather_observations quando disponível.

### Phase 10: Onboarding Wizard

**Goal**: Novo usuário que faz login pela primeira vez é guiado por um wizard de 3 passos e sai com calendar conectado e push ativo
**Depends on**: Phase 9
**Requirements**: UX-04
**Plans**: 2 plans

Plans:
- [x] 10-01: Backend — migration onboarding_done + auth.js + usuarios.js (PATCH /me) + app.js + api.js (UX-04)
- [x] 10-02: Frontend — CSS wizard + abrirWizard() + fecharWizard() + ativarPushNotificacoes() + disparo em carregarSessao() (UX-04)

**Details:**

10-01: Migration idempotente `onboarding_done INTEGER NOT NULL DEFAULT 0` via try/catch ALTER TABLE. GET /api/auth/me ampliado para incluir `onboarding_done`. Novo router usuarios.js com PATCH /api/usuarios/me protegido por requireAuth. Validação strict (onboarding_done !== 1 → 400). UPDATE usa req.session.userId nunca req.body.userId (T-10-01). Mount em app.js.

10-02: Modal wizard injetado por JS (não HTML estático) para evitar exibição antes da sessão carregar. 3 passos: (1) Introdução, (2) Conectar Calendar, (3) Ativar Push. Focus trap via keydown/Tab. Shake animation ao clicar fora ou Escape (não fecha acidentalmente). `fecharWizard()` chamado antes do PATCH — degradação silenciosa se PATCH falhar. Disparo em `carregarSessao()` se `!usuario.onboarding_done`.

## Milestone Summary

**Decimal Phases:**
- Phase 09.1: Integração Dados Meteorológicos Reais (inserida após Phase 09 para integrar fontes reais antes do onboarding)

**Key Decisions:**

| Decisão | Justificativa |
|---------|---------------|
| stale=true threshold de 120min | 2x o intervalo do cron — tolera falhas transitórias sem alarme falso |
| Best-effort pattern no auto-trigger CSV | Import stats sempre retornados mesmo se recálculo falhar; recalculo field sinaliza status do side-effect |
| Modal wizard injetado por JS | Evita exibição antes da sessão ser carregada; sem markup estático que apareceria no flash inicial |
| fecharWizard() antes do PATCH | Degradação silenciosa — UX não quebra se PATCH falhar |
| Visual Crossing + REDEMET paralelos no scheduler | Independentes; sem dependência de ordem; falha de um não bloqueia o outro |
| Fix CR-01 em visualCrossing | Janela de datas usava wall-clock -24h → buggy em meia-noite; corrigido para dia-calendário anterior |

**Issues Resolved:**

- Progresso do modal de onboarding não persistia após refresh — corrigido com PATCH /me gravando flag antes de fechar

**Issues Deferred:**

- UX-01 (seletor de antecedência) não implementado como feature nova — verificado que o campo `alert_threshold` já existia desde v1.0; o seletor já estava funcional. Plano 09-03 serviu como smoke test, não implementação.
