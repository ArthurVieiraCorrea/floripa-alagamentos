# Code Review: Phase 09.1

## Summary

Reviewed five files implementing phase 09.1: new `weather_observations` table DDL, `visualCrossingService.js`, `redemetService.js`, scheduler integration, and `.env.example` documentation. The overall implementation is solid — no-throw discipline (D-10) is consistent, INSERT OR IGNORE (D-12) is correct, the D-11 UPDATE-after-COMMIT ordering is respected, and neither service leaks API keys through logs. Four warnings were found, none of which causes data loss or crashes, but two (CR-01 and CR-02) will produce subtly wrong data under specific timing conditions that occur regularly.

---

## Findings

### CR-01 | WARNING | visualCrossingService.js

**Issue:** The `toLocalDate()` helper subtracts 3 hours from UTC to simulate UTC-3, then calls `.toISOString().substring(0,10)`. This is correct for the date portion. However, `yesterday` is computed as `now - 24h` in wall-clock time, not calendar-day yesterday. When `now` is between 00:00 and 02:59 UTC (i.e., between 21:00 and 23:59 Florianópolis time the previous calendar day), `toLocalDate(yesterday)` returns the date *two* calendar days ago, and `toLocalDate(now)` returns *yesterday*. The API window then covers only roughly 24h but spanning entirely wrong calendar days (e.g., 21:00 on Apr-20 → 21:00 on Apr-21 when actual local time is 23:55 on Apr-21). All hours between 22:00–23:59 local on the current day are missed.

**Line:** 42–51 (`buildUrl()`)

**Impact:** Between 21:00 and 23:59 Florianópolis local time (00:00–02:59 UTC), every hourly call returns data from the wrong day-pair. The current local evening hours — which are the highest-risk window for flooding events — are excluded from the observed precipitation total written to `forecasts_meta.precip_48h_mm`.

**Recommendation:** Use `toLocalDate(now)` as `date2` (correct) but compute `date1` as the calendar day *before* `date2` rather than 24 wall-clock hours earlier:
```javascript
function buildUrl() {
  const KEY = process.env.VISUAL_CROSSING_API_KEY;
  const now = new Date();
  // Shift to local (UTC-3) to get correct calendar date
  const localNow = new Date(now.getTime() - 3 * 60 * 60 * 1000);
  const date2 = localNow.toISOString().substring(0, 10); // today in local TZ

  // Subtract one calendar day from the local date string
  const prevDay = new Date(localNow.getTime() - 24 * 60 * 60 * 1000);
  const date1 = prevDay.toISOString().substring(0, 10); // yesterday in local TZ

  return [...].join('');
}
```
This is semantically equivalent but avoids the midnight-window bug because both `date1` and `date2` are derived from the same UTC-shifted anchor.

### CR-02 | WARNING | redemetService.js

**Issue:** `registro.validade_inicial` is accessed directly without a null/undefined guard. If the REDEMET API returns a registro where `validade_inicial` is absent or null (e.g., a partial response or schema change), `registro.validade_inicial.replace(...)` throws `TypeError: Cannot read properties of null (reading 'replace')`. This error is thrown inside the transaction loop, causing a `ROLLBACK` and loss of all METARs in the current batch.

**Line:** 129 (`const utcTime = new Date(registro.validade_inicial.replace(' ', 'T') + 'Z')`)

**Impact:** A single malformed METAR record aborts the entire transaction. No METARs from the current fetch cycle are persisted, even if the other records were valid. This is recoverable (next cron tick retries), but any single-hour METAR with a missing `validade_inicial` field will perpetually block persistence for all concurrent records in that fetch.

**Recommendation:** Guard the field access and skip invalid records individually:
```javascript
if (!registro.validade_inicial) {
  console.warn('[redemet] registro sem validade_inicial — ignorado');
  continue;
}
const utcTime = new Date(registro.validade_inicial.replace(' ', 'T') + 'Z');
```

### CR-03 | WARNING | visualCrossingService.js

**Issue:** `totalMm` accumulates the sum of all hourly `precip` values across *both* days returned by the API, but the field it writes to is named `precip_48h_mm`. With a 24h query window this is semantically wrong in the other direction: the value stored will represent at most ~24h of observed precipitation but will be read by `riskEngine.js` as if it were a 48h figure. More critically, if the API returns two full calendar days (48 hour slots), `totalMm` will sum all 48 values and write a 48h total into `precip_48h_mm` — which at least is labeled correctly. But during the midnight edge case described in CR-01, the window may span only 24 hours of a non-current day, making the stored value misleading in both directions.

**Line:** 120–143 (the accumulation loop inside `fetchAndCacheObservations()`)

**Impact:** `forecasts_meta.precip_48h_mm` may hold a 24h sum labeled as 48h (under-representing risk) or a stale-day sum (wrong temporal context). The riskEngine reads this field without knowing the source window, so miscalibrated values directly affect risk scores.

**Recommendation:** The RESEARCH.md (Pattern 4) correctly notes that the right approach is to query `weather_observations` *after* the INSERT transaction, summing only the rows within the desired lookback window — rather than summing the loop counter. This eliminates the double-day ambiguity and the midnight edge case:
```javascript
// After COMMIT:
const row = db.get(
  `SELECT COALESCE(SUM(precipitacao_mm), 0) AS total_mm
     FROM weather_observations
    WHERE bairro = 'florianopolis'
      AND fonte = 'visual-crossing'
      AND observed_time >= datetime('now', '-3 hours', '-24 hours')
      AND observed_time <  datetime('now', '-3 hours')`
);
const rounded = Math.round((row.total_mm ?? 0) * 10) / 10;
db.run(`UPDATE forecasts_meta SET precip_48h_mm = ? WHERE id = 1`, [rounded]);
```
This was the approach documented in `09.1-RESEARCH.md` but was not carried into the implementation.

### CR-04 | WARNING | redemetService.js

**Issue:** The REDEMET URL is built in `buildUrl()` with the API key interpolated directly as `?api_key=${KEY}`. The URL string is then passed as a parameter to `fetchWithRetry(url)`. If `fetchWithRetry` ever logs the URL on error (it currently does not, but future maintainers may add logging there), the API key would be exposed. More concretely, Node.js `fetch` request failures sometimes include the URL in the `Error.message` of `TypeError: fetch failed` (e.g., when DNS resolution fails), and that error message is logged at line 156: `console.error(\`[redemet] Falha apos... ${err.message}\`)`. A DNS failure would log the full URL including the key.

**Line:** 48 (`buildUrl()`), 156 (outer catch `console.error`)

**Impact:** If the REDEMET endpoint is unreachable (DNS failure, network partition), the API key leaks into application logs via `err.message`. This is the same pattern present in `visualCrossingService.js` line 162 for the Visual Crossing key.

**Recommendation:** Scrub the key from error messages before logging, or restructure `fetchWithRetry` to accept a separate display-safe URL for logging:
```javascript
// Option A: strip key from error message before logging
const safeMsg = err.message.replace(process.env.REDEMET_API_KEY ?? '', '[REDEMET_KEY]');
console.error(`[redemet] Falha apos ${MAX_RETRIES} tentativas: ${safeMsg}. Fonte ignorada.`);
```
Apply the same fix to `visualCrossingService.js` line 162.

### CR-05 | INFO | visualCrossingService.js

**Issue:** The `buildUrl()` function reads `process.env.VISUAL_CROSSING_API_KEY` directly at call time. If `KEY` is `undefined` (which can happen if `fetchAndCacheObservations()` is called without the guard running first, e.g., via a direct unit test), the URL becomes `...&key=undefined`, which causes an HTTP 401 from Visual Crossing. The guard at the top of `fetchAndCacheObservations()` prevents this in normal runtime, but `buildUrl()` is not marked as internal-only and has no own guard.

**Line:** 40 (`const KEY = process.env.VISUAL_CROSSING_API_KEY`)

**Impact:** Low in production (guard is always reached first). Confusing in tests — the function produces a syntactically valid but semantically broken URL without warning.

**Recommendation:** Either add an assertion in `buildUrl()` (`if (!KEY) throw new Error(...)`) so failures are explicit, or keep `buildUrl()` as a purely private helper with a JSDoc `@private` annotation and a comment that it must only be called after the guard.

### CR-06 | INFO | redemetService.js

**Issue:** The `data?.status` check at line 102 treats `status: false` (which the REDEMET API returns for API errors such as `API_KEY_MISSING`) as a falsy value and emits a generic `[redemet] Resposta inesperada:` warning. While the behaviour is correct (return without throw), the log message will include the full JSON body `{"status":false,"message":"API_KEY_MISSING",...}` which does not contain the key value but may confuse operators into thinking the service is broken rather than misconfigured.

**Line:** 102–105

**Impact:** Operator confusion in development. If the key is set but invalid, this is the only signal — and it's not clearly differentiated from a network error.

**Recommendation:** Check `data.status === false` separately and log a more descriptive message:
```javascript
if (!data?.status) {
  const reason = data?.message ?? 'unknown';
  console.warn(`[redemet] API retornou status=false (razao: ${reason}) — fonte ignorada`);
  return;
}
```

### CR-07 | INFO | scheduler.js

**Issue:** The startup IIFE wraps each call in its own `try/catch` to ensure one failure does not prevent subsequent calls. However, `fetchVisualCrossing()` and `fetchRedemet()` already follow the no-throw pattern (D-10), so the surrounding `try/catch` blocks in the IIFE at lines 68–75 are redundant. They add noise without harm, but they could mask unexpected throws from future refactoring that accidentally introduces a throw path.

**Line:** 62–86 (startup IIFE)

**Impact:** None at runtime. The extra try/catch blocks are belt-and-suspenders — correct but unnecessary for D-10-compliant services.

**Recommendation:** Keep the outer try/catch only for `fetchAndCacheForecasts()` and `calcularRiscos()` (which may not guarantee no-throw), and add a comment clarifying that `fetchVisualCrossing()` and `fetchRedemet()` are already no-throw:
```javascript
// D-10: fetchVisualCrossing e fetchRedemet são no-throw por contrato
await fetchVisualCrossing();
await fetchRedemet();
```
This is a style issue only — the current code is functionally correct.

## Verdict

**PASS WITH WARNINGS**

The implementation is functionally correct for the happy path and meets D-08, D-09, D-10, D-12, and the scheduler ordering (D-15/D-16). No critical bugs were found. Four warnings require attention before production traffic:

- **CR-01** (midnight window bug in `buildUrl`) and **CR-03** (totalMm accumulation vs. DB-query approach) are the highest-priority fixes — they produce subtly wrong `precip_48h_mm` values during the 21:00–23:59 local window (exactly when evening storms occur).
- **CR-02** (null guard on `validade_inicial`) is a straightforward defensive fix that prevents a potential batch rollback.
- **CR-04** (API key in `err.message`) should be fixed before enabling these services in any environment where logs are aggregated or forwarded.

CR-05, CR-06, and CR-07 are informational and can be addressed opportunistically.
